Omni UI Design Guide

The complete design system for building screens in the Omni clinical workspace. All measurements, colors, type sizes, component patterns, and layout rules are defined here. Follow this guide exactly when creating new views or extending existing ones.

Rules

Use Space Grotesk for: patient names, card titles, section labels, form labels, KPIs, badge text, nav labels, all numeric displays, keyboard hints.
Use Open Sans for: form values (contenteditable), note bodies, table cell data, body prose, placeholder text, status bar text.
Base font-size: 14px on html element. All rem/em values cascade from this.
Minimum readable size: 10px — only for uppercase labels and badges. Never use below 10px.

Spacing

4px base unit. All spacing is a multiple of 4. The goal is maximum density without crowding — prefer 8px and 12px gaps for most UI, 4px for within-component micro-spacing.

Density Rules

This is a dense, operator-focused UI. The goal is maximum information in the viewport with no vertical wasted space. Every sizing decision should bias toward compact.

✓ Do
  • Use 12–13px for most body content
  • Prefer 6–8px row padding in tables
  • Show data inline, not in modal sheets
  • Use segmented controls, not full tab bars
  • Stack multiple data points in single rows
  • Use 10px uppercase labels (not 12px)
  • Truncate with ellipsis + tooltip rather than wrapping
  • Put quick actions in the header row of a card
  • Use status dots in tables, not full badge pills
✗ Don't
  • Use font sizes below 10px anywhere
  • Add excessive vertical padding between sections
  • Wrap text in table cells — truncate instead
  • Use decorative empty states with illustrations
  • Add margin between every single component
  • Use full badge pills in dense tables
  • Leave large empty areas "for breathing room"
  • Put confirmation dialogs for routine actions
  • Use more than 4px gap between label and value

Interaction & Animation

Standard transition 0.15s ease — all hover state changes (background, color, border)
Nav expand 0.2s cubic-bezier(.4,0,.2,1) — slightly more physical
Palette open 0.15s ease-out with translateY(-8px) → 0 + scale(0.98) → 1
Label fade (nav) opacity 0 → 1, 0.15s
No bounce, no spring physics, no delayed entrances. Everything is immediate and precise. Clinical tools require predictable interfaces.
No page transitions. Content swaps are instant. Only overlay appearances get animation.

Design Principles

1. Density over whitespace. This is a workplace tool used 8+ hours a day. Staff need to see maximum data, not breathe. Every pixel should earn its place.
2. Keyboard-first, mouse-second. The command palette, keyboard shortcuts, and segmented controls all reduce mouse travel. Design new patterns with keyboard access as the primary path.
3. Inline editing, not modal sheets. Patient records use contenteditable fields. Prefer editing in context over opening forms in modals wherever practical.
4. Status is always visible. Patient status, connection health, location, and current context (breadcrumb) are always on screen. Never make staff hunt for state.
5. Warm neutrals, not cold grays. The surface palette has a warm, slightly cream/tan undertone. This makes the interface feel less clinical and reduces eye strain over long sessions.
6. One primary action per context. Each panel or card has one clearly weighted primary button. Secondary actions are ghost or icon-only.
7. Consistent token usage. Never hardcode hex values. Always use CSS custom properties. This enables future theming and ensures visual consistency across all screens.
8. Preserve the shell. Left nav, top bar, status bar, and patient rail are invariant. Content panels change; the shell does not. New views are always content panel replacements, not full-page replacements.
